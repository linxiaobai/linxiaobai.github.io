优化细节

count(column)和count(*)

在mysql数据库预热时，每张表记录的总数就被记录在一个地方，所以查询表的总记录时，count(*)更优

跟在where后面的条件，like等不确定条件放在后面，=的条件放在前面。

因为like等放在前面，后面的所有字段的索引都将不能使用

减少join操作

因为join在mysql的实现机制是要将临时结果集放在临时表中（其中一种算法是这样的），
所以比如a left join b limit 0,10，需要将a的结果集和b的结果集合并放在一张临时表中，然后进行操作。
所以可以这样优化
(a limit 0,10) left join (b limit 0,10) limit 0,10
这样，至少减少了中间结果集的记录数，一定程度上优化了性能，但还是尽量少用，因为mysql中join的性能处理远不及oracle强大

避免使用select *

涉及到order by的排序实现
大致分为两种：
有序索引排序：执行计划中的type为range、ref、index
排序算法排序：执行计划中成为filesort，由于没有可以利用的有序索引取得有序的数据，MySQL只能通过将取得的数据在内存（sort buffer）中进行排序然后再将数据返回给客户端
1.首先根据相应的条件取出相应的排序字段和可以直接定位行数据的行指针信息，然后在 sort buffer 中进行排序(Mysql 4.1之前)
2.一次性取出满足条件行的所有字段，然后在 sort buffer 中进行排序。(Mysql 4.1之后)
【注】
max_length_for_sort_data大则用第二种算法，小则用第一种算法
如果数据库出现大量的排序等待，需要增大max_length_for_sort_data的值

减少排序，排序很消耗CPU

可以尽量用索引来减少排序
